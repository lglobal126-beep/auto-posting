## 프로젝트 개요

- **프로젝트 이름**: 맛집 인플루언서 자동 포스팅 도우미
- **목표**: 사용자가 식당 방문 후 사진·영상·영수증만 업로드하면, 네이버 블로그용 리뷰 글과 인스타그램용 캡션을 자동 생성하고, 연동된 계정으로 자동 업로드까지 수행하는 개인용(2인) 도구를 만든다.
- **타깃 사용자**: 맛집 인플루언서 지망생 1명 + 지인 1명 (총 2명)
- **플랫폼**: 웹앱(PWA) 기반, 모바일 브라우저에서 사용

## 핵심 기능

1. **계정 및 연동**
   - 이메일/비밀번호 기반 로그인 (Supabase Auth)
   - 네이버 블로그 계정 연동 (OAuth)
   - 인스타그램(Instagram Graph API) 계정 연동 (OAuth)

2. **콘텐츠 업로드**
   - 사진/영상 업로드
   - 영수증 이미지 업로드
   - 간단 메모 입력 (방문 목적, 맛/서비스/분위기 메모 등)

3. **정보 추출 및 가공**
   - 영수증 OCR을 통한 텍스트 추출 (Tesseract)
   - 영수증 텍스트에서 식당 이름, 메뉴, 가격, 방문 시간 등 파싱
   - 영수증에서 인식한 상호명을 기반으로 네이버 지도/검색/리뷰 정보를 조회
   - 네이버 리뷰 텍스트와 장소 정보를 바탕으로 운영시간, 주차, 편의시설, 분위기, 인기 메뉴 등을 자동 수집

4. **AI 기반 문장 생성**
   - LLM API를 사용해 네이버 블로그용 제목/본문 생성
   - 인스타그램용 짧은 캡션 + 해시태그 리스트 생성
   - 톤 & 매너: 1,000만 구독자급 맛집 인플루언서 스타일, 솔직하지만 세련된 존댓말

5. **초안 관리 및 발행**
   - 생성된 초안(블로그 글/인스타 캡션) 미리보기 및 수정
   - 네이버 블로그/인스타그램에 각각 또는 동시에 발행
   - 발행 결과(게시물 URL, 상태) 조회

6. **운영 조건**
   - 클라우드 무료 플랜을 활용하며, 월별 무료 쿼터 내 사용을 전제로 설계
   - 사용자 수는 최대 2명으로 가정

## 기술 스택

- **프론트엔드**
  - Next.js (React, TypeScript)
  - 배포: Vercel 무료 플랜
  - 인증/데이터 연동: Supabase JS SDK

- **백엔드**
  - Python FastAPI
  - 배포: Render 무료 Web Service (또는 유사한 무료 호스팅)
  - 역할: 영수증 OCR, LLM 호출, 네이버/인스타 연동, 비즈니스 로직 처리

- **데이터/스토리지**
  - Supabase (Postgres, Auth, Storage)
  - 초안/포스트/토큰/메타데이터 저장

- **AI/외부 서비스**
  - LLM API (월별 무료 쿼터 제공 서비스; 실제 키는 사용자 환경에 설정)
  - Tesseract OCR (서버 컨테이너 내 설치)
  - 네이버 블로그 Open API
  - 인스타그램 Graph API

## 요구 사항 정리

1. **비즈니스 요구 사항**
   - 사용자는 글쓰기 부담 없이 사진·영수증만으로 고품질 리뷰를 얻을 수 있어야 한다.
   - 생성된 문장은 과장 광고가 아닌, 솔직한 후기 느낌을 유지해야 한다.
   - 네이버 블로그와 인스타그램 모두에 올릴 수 있어야 하며, 둘 중 하나만 선택 발행도 가능해야 한다.

2. **기능 요구 사항**
   - [FR-1] 사용자는 이메일/비밀번호로 로그인/로그아웃 할 수 있다.
   - [FR-2] 사용자는 네이버 블로그 계정을 OAuth로 연동/해제할 수 있다.
   - [FR-3] 사용자는 인스타그램 계정을 OAuth로 연동/해제할 수 있다.
   - [FR-4] 사용자는 사진/영상/영수증을 업로드하고 메모를 입력할 수 있다.
   - [FR-5] 시스템은 영수증으로부터 텍스트를 추출하고, 식당/메뉴 정보를 파싱할 수 있다.
   - [FR-6] 시스템은 LLM을 사용해 네이버 블로그용 글과 인스타그램용 캡션을 생성할 수 있다.
   - [FR-7] 사용자는 생성된 초안을 수정한 뒤 저장할 수 있다.
   - [FR-8] 사용자는 초안을 네이버/인스타에 발행하거나, 둘 중 하나에만 발행할 수 있다.
   - [FR-9] 시스템은 발행 결과(성공/실패, URL)를 저장하고, 사용자가 조회할 수 있게 한다.

3. **비기능 요구 사항**
   - [NFR-1] 서비스는 2명 사용자를 대상으로 하며, 월별 무료 쿼터 내에서 동작해야 한다.
   - [NFR-2] 첫 요청 시 슬립에서 깨우는 딜레이(수 초)는 허용한다.
   - [NFR-3] 민감 정보(OAuth 토큰, LLM 키)는 서버 환경 변수 또는 안전한 저장소에 보관한다.
   - [NFR-4] 초기 버전에서는 모바일 브라우저 사용성을 우선 최적화한다.
